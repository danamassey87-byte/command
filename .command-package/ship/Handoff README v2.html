1 · Who this is for

You are an AI coding agent (Antigravity, Claude Code, or similar) or a human engineer building Command — a single-tenant real-estate ops platform for Dana Massey, a solo agent in Illinois. One user. One workspace. No multi-tenant logic, no user-management UI, no role-based permissions. Build for speed and clarity, not scale.

Single-tenant means — hardcode Dana as the default user. Skip org/team abstractions. Every entity has owner_id but it's always the same UUID for now. When we add a team later (Jamie + VAs), we'll lift that constraint, not rebuild.

2 · What we're building

Command is Dana's daily operating system. It replaces ~12 current tools by wrapping the 3 she keeps (Lofty, Transact, Blotato) and consolidating everything else. Core surfaces:

  • Mobile: Dashboard, Prospecting, Tasks + Notes, Buyer Showings, Showing Route
  • Desktop: CRM, Email Builder, Lead Sources, Inspo Vault, Content Studio, Listing Appt + CMA, Deals, KPI, Bio Link, Properties, Money, Checklists, Mentor, Seller Updates, Agreements, Vendors, Social Research, Content Publishing, Local Places & Events, OH QR, Library, Studio (b-roll composer)
  • Settings / Admin: Usage & Costs, System Health, Integrations hub (with detail pages for Lofty, Transact, Compliance, Slack)

See the wireframes HTML for every screen. The Data Model doc is the authoritative schema.

v2 additions. Section 17 (Feature modules) covers the 8 deep-dive surfaces added after v1: Relationships+ · Home-value microsite · Team & permissions · SEO/AEO · Reviews & referrals · Print & delivery · Expired/FSBO · Post-close. Phase 1–3 are unchanged — modules ship in Phases 4–6 per the roadmap below.

3 · Non-goals

  • Multi-tenant SaaS. We are not selling Command.
  • Native apps. It's a responsive web app + iOS Safari "add to home screen." Push is via Slack.
  • Replacing Lofty, Transact, or Blotato — we wrap them.
  • MLS API. Until Dana has a paid MLS data provider, the MLS fields stay nullable and populated by AI research + manual entry.
  • Billing / subscription UI.

7 · Slack in detail

App scopes

Scopes needed at install: chat:write, chat:write.customize, channels:read, channels:manage, groups:read, files:write, users:read, commands, reactions:write, im:write.

Channel naming convention

Matches Dana's existing Slack sidebar format exactly:

  • 🤍 SELLERS group — #seller_firstname-lastname or #seller_firstname-lastname_### (address suffix optional)
  • 🖤 BUYERS group — #buyer_firstname-lastname
  • 🎯 LEADS & OPPS group — #expired-listings, #open-house-opportunities, #openclaw
  • 🔥 GENERAL SHIZ group — #all-dana-massey-realtor, #cma-pricing-strategies, #client-updates_airtable, #signage, #daily-brief, #content-review, #compliance, #system, #wins

Channels are created via conversations.create when a new buyer/seller is added. Dana is invited. Bot joins. Archive 30d after deal closes (conversations.archive) — not deleted, searchable.

Event routing

Every external event hits /api/webhooks/{source}, which looks up a row in notification_rules and posts to the matching channel(s). Rules are user-editable in Admin · Slack. Example shape:

{
  source: "transact",
  event: "bba_signed",
  destination: "#buyer_{slug} · #seller_{slug}",     // templated
  payload_template: {                                 // Slack Block Kit
    blocks: [
      { type: "section", text: { type: "mrkdwn",
        text: ":page_facing_up: *BBA signed* by {contact.name} ({property.address})" } },
      { type: "context", elements: [{ type: "mrkdwn", text: "via Transact · {signed_at}" }] }
    ]
  },
  interactive_actions: [
    { text: "✓ Mark done", value: "complete_step", step: "bba_signed" },
    { text: "⏱ Snooze 2h",  value: "snooze", duration: "2h" },
    { text: "↗ Open in Transact", url: "{transact.url}" }
  ]
}

Interactive callbacks

Slack button clicks POST to /api/slack/interactive. Verify signing secret. Route to the matching handler by value. Update the original message with response_url to show the result inline ("✓ marked done · 2m ago").

Daily brief

Inngest cron at 7:00 America/Chicago Mon–Sat. Reads today's schedule, overdue follow-ups, hot signals, KPI delta, weather for any OHs. Assembles a Block Kit message and posts to #daily-brief. 5 PM wrap-up summarizes the day and queues tomorrow.

Privacy on lockscreen

When redact_client_names is ON, every notification goes out as text: "New activity on a buyer file" at the Slack notification level and puts the real content in attachments (which Slack mobile shows only after unlock). Implement via the text vs blocks split.

12 · Cost controls

Every LLM call must pass through lib/ai/bill.ts. This wrapper: enforces a per-request max-tokens cap, logs token + $ to cost_ledger, checks the monthly budget, and rejects if the monthly cap is exceeded.
  • Budgets are per-service rows in cost_ledger. Default $600/mo total across all AI services.
  • 80% → post to #system. 95% → post + @dana. 100% → reject with friendly error.
  • Prefer Haiku for classification, titling, one-liners. Sonnet only when output matters.
  • RAG queries pull summary cards before chunks — cuts retrieval tokens ~80%.

13 · Compliance pipeline

Every outbound content item (social, email, listing description, Bio Link copy) passes through lib/compliance/check.ts:

  1. Classify the content type (property-specific / general-real-estate / personal).
  2. Retrieve matching rules from Pinecone compliance index (RESPA, Fair Housing, Illinois license act, NAR guidelines, Dana's brokerage handbook).
  3. Run Sonnet with cited rules → verdict: pass · warn · block.
  4. Log to compliance_checks with cited chunks.
  5. If block, post to #compliance with override link. Dana can override with a reason, which gets logged to the audit trail.

14 · Testing

  • Vitest for unit + integration (DB-backed, against a local Supabase branch).
  • Playwright for a small e2e set: login → create contact → log interaction → send Slack notification (mocked) → open OH → run kiosk sign-in.
  • Every integration has a __fixtures__/ folder with canned API responses for deterministic tests.
  • Slack interactive callbacks have a dedicated suite — signature verification is easy to break.

15 · Deploy

  • Web: Vercel (production + preview per PR).
  • Jobs: Inngest cloud (self-hosting later if cost wins).
  • DB: Supabase (prod = hosted project; dev = local supabase CLI).
  • Secrets: Vercel env per branch.

Updating an existing app, not greenfield

Command is replacing/extending Dana's current app in the same Supabase project. Before writing migrations, the agent must:

  1. Run supabase db pull against the existing project to capture the current schema.
  2. Diff it against Data Model.html. For each missing table/column: write an additive migration. Do not drop or rename existing tables without Dana's explicit approval — they may have live data.
  3. Existing tables that overlap (contacts, deals, etc.) — extend rather than replace. Add Command-specific columns with defaults.
  4. Log every schema change to /docs/migrations-log.md with a one-line rationale.

17 · Feature modules (v2)

Each module maps to a wireframe section + its v2 Data Model entities + external integrations. Two-paragraph ship plans. Read alongside Data Model v2.html.

17.1 · Relationships+ · social & life events

Scope: 9-platform social profile tracking (IG, FB, TT, LI, X, YT, Threads, Pin, Nextdoor) · family links (spouse/kids/pets) · sphere-wide 90-day life-event feed. Wireframes: desktop · relationships (7 screens). Entities: SocialProfile, FamilyLink, LifeEvent.

Integrations: IG/FB/LI via Graph API where Dana has business auth. TT/X/YT/Threads/Pin/Nextdoor are paste-URL + AI verification — no scraping, no stored credentials. Match confidence ≥0.9 sets verified = true. Life events generated nightly from birthdays (contact field), anniversaries (FamilyLink.since), closing dates (Deal.closed_at), and observed social signals. Outreach suggestions hit the daily brief.

17.2 · Home-value microsite

Scope: Public AVM landing · 4-source blend with agent adjusts · 14-touch drip · seller-lead dashboard · embeddable widget · funnel attribution. Wireframes: desktop · homevalue (9 screens). Entities: Valuation, SellerLead, MicrositeDrip, MicrositeDripStep, MicrositeDripRun.

Integrations: Zillow (GetEstimate), Redfin (scraped-with-license), Realtor (RapidAPI), Attom (paid). Cache 24h per property. Blend at read-time; never persist the blend. Drip runs on Inngest. Public landing is a separate Next.js route group (app/(public)/estimate) deployed on same domain. SEO-critical — server-side render with proper JSON-LD (shares SchemaFragment infra from 17.4).

17.3 · Team & permissions

Scope: 6 role templates (Agent, TC, VA, ISA, Showing-only, Custom) · 28 granular permissions · per-deal scope overrides (time-boxed) · 7 lead-routing rules · co-listing with 5 participants + GCI waterfall · delegation inbox · 90-day audit log. Wireframes: desktop · team (10 screens). Entities: TeamMember, Role, Permission, ScopeOverride, RoutingRule, CoListing, GCISplit, DelegationItem, AuditEvent.

Integrations: Stripe for seat billing (tier flag on team_members). All mutations flow through a permission middleware (lib/team/guard.ts) — never query raw DB from a route handler without the guard. Audit log is append-only; writes go to both Postgres and R2 (cold storage) on a 7-day rollover for SOC2-ready retention. Lifts the owner_id = Dana constraint from v1; introduces workspace_id + actor_id on every row via additive migration.

17.4 · SEO & AEO

Scope: Keyword ranks · AI citation share-of-voice across 4 LLMs · hub-and-spoke topical map · blog editor with SEO+AEO scoring · GMB composer · JSON-LD coverage matrix · weekly visibility audit. Wireframes: desktop · seoaeo (8 screens). Entities: SeoKeyword, RankingSnapshot, AICitation, HubPage, SpokePage, GmbPost, SchemaFragment, SeoAudit.

Integrations: DataForSEO (or SerpAPI) for ranks — weekly batch. Google Business Profile API for GMB (scope business.manage); scheduled posts enforce a 10-minute cancel window. AI citation polling is a custom Inngest job hitting ChatGPT (with-search), Claude, Perplexity, and Gemini weekly for 40 configured queries — full response bodies cached in R2 for audit. Schema validator is local (json-schema-of-json-ld) — no external call.

17.5 · Reviews & referrals

Scope: Automated review asks on post-close trigger · multi-platform review ingest · referral attribution chain. Wireframes: desktop · reviews (9 screens). Entities: ReviewRequest, Review, Referral.

Integrations: Google Business Profile API (reviews:read + reply:write), Zillow/Realtor via scheduled scrape where public, Facebook via Graph. Review requests fire from post_close_touches (module 17.8) at +7d. Referral attribution links back to Contact on both ends; GCI field populated on deal close. Thank-you flow uses Print & Delivery (17.6) for handwritten-note sends.

17.6 · Print & delivery

Scope: Postcards, letters, handwritten notes, pop-by tags · provider-agnostic adapter · batch send with per-recipient tracking. Wireframes: desktop · printdeliv (8 screens). Entities: PrintOrder, DeliveryBatch.

Integrations: One adapter interface (lib/integrations/print/adapter.ts) implemented by Thanks.io, Handwrytten, and Sendoso. Address validation via USPS API (free tier) before submission. Cost pushed to cost_ledger on submit — visible in the same dashboard as AI spend. Delivery-status webhooks update per-recipient rows; failures surface in #system.

17.7 · Expired + FSBO prospecting

Scope: Expired/FSBO lead pipelines · 14-day cadence · live call coach with transcript · AI-personalized letters · TCPA/DNC compliance gate. Wireframes: desktop · expiredfsbo (9 screens). Entities: ExpiredLead, FSBOLead, CallOutcome.

Integrations: Data sources are pluggable — Vulcan7, RedX, Landvoice, or MLS-expired feed. Dialer via Twilio Voice. Live transcription via Deepgram streaming; coach prompts run against the transcript buffer in 5s windows. Hard gate: no dial happens without dnc_result = 'clear' from the FTC DNC registry check (daily-refreshed cache). TCPA consent tracking in the same row.

17.8 · Post-close · Client for life

Scope: Auto-created 5-year touch plan per closed deal · anniversary triggers · home-owned alerts · scheduled review + referral asks. Wireframes: desktop · postclose (9 screens). Entities: ClientForLifePlan, PostCloseTouch.

Integrations: Plan instantiates on Deal.stage → closed. Template picks based on client type (standard, investor, sphere-vip). Touches scheduled as Inngest delayed jobs up to 5 years out — safe because Inngest handles durable long-running waits. Reaches into Reviews (17.5), Print (17.6), and Email (from v1) for execution channels. Opt-out flag on the plan cascades cancellation of all future touches.

17.9 · Lead sources & pay-at-close attribution

Scope: First-class modeling of every PAC (pay-at-close) and subscription lead source — seeded with CertiLeads, ClosingLeads, AtClosing, Movoto, Realty.com, LeadDeck, FastCashOffers. Per-source buyer % / seller %, attribution window (per-deal · 1yr · 2yr · lifetime), intake method, live ROI dashboard, pipeline-fee forecast. Dedicated cash-offer-first routing rule through FastCashOffers. Wireframes: desktop · leadsources (4 screens). Entities: LeadSource, LeadAttribution, ReferralFee, CashOfferRouting.

Integrations: Intake per source is either email-forwarder parsing (leads@<vendor>.com → Command parser), Zapier/Make webhook, or API polling where vendor exposes one (CertiLeads/Movoto). Fee % is locked on LeadAttribution at arrival — changes to LeadSource.buyer_pct/seller_pct do not retroactively affect in-flight contacts. ReferralFee rows auto-create as deals move to under-contract (status=suggested) and lock on close (status=confirmed); user can override with a required override_reason. The cash-offer router fires on tag/form-intent match, submits to FastCashOffers, waits up to 24h SLA, then on decline auto-enrolls to a fallback sequence. Fees always debit the agent split unless who_pays is overridden. Fee surfaces on the Deal Room sidebar (projected GCI → referral line → net pre-split) and gates the "mark closed" action until confirmed or waived with note.

18.1 · Quick actions (header, top-right)

Two always-available actions, docked in the top header to the left of the Edit / Settings / Sign-out cluster. Not a floating FAB stack in the bottom-right — that pattern is deprecated and must be removed wherever it exists.

Dedup — remove the third button. The current live app shows three bottom-right FABs (note · tasks · note). The third is a duplicate of the first and must be deleted. One Quick Note entry point only.

Placement & behavior

  • Position: header right rail, order: [+ Quick Add] · [🔔] · [✎ Note] · [✓ Tasks] · [date/time] · [Edit] · [⚙] · [Sign out]. Icon-only at ≤1280px, icon+label above.
  • Mobile: collapsed into the top app-bar overflow menu; accessible from every route, not just Home.
  • Drawer: right-side slide-in (360px desktop, full-width mobile). Esc / overlay-click dismisses. Persists draft in localStorage so navigating away doesn't lose input.
  • Keyboard: N opens Quick Note, T opens Tasks, from anywhere that's not a text field.
  • Scope: global — present on Home, Prospect, CRM, Deals, Content, Email, Bio Link, Money, Tool Kit, and every detail view. No exceptions.
